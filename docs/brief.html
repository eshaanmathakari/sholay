v0.2 · macOS · review-gated

A small agent that operates your Mac the way a person does — and asks before it does anything risky.

Built on Anthropic's Computer Use API. Designed for desktop tasks, not web pages — opens apps, types into native forms, navigates Finder, installs from a .dmg, plays local media. A human reviewer sees and approves any action that could change your machine.

See how it works
01 / what it is

An agent loop with three checkpoints between the model and your machine.

Anthropic's Computer Use API lets a model take screenshots, decide one action, run it, and see the result. This project wraps that loop with the parts you actually need to run it on a real Mac: a friendly interactive prompt, reliable host primitives for the brittle stuff, and three review checkpoints so nothing irreversible happens without a person seeing it.

Sees the screen

The model receives a screenshot before every decision. Retina pixels are scaled down so it thinks in display points, not raw resolution. It narrates what it sees, then picks one action.

Two kinds of tools

Vision-driven actions (click, type, drag) handle most UI work. A small set of reliable Mac primitives covers the brittle stuff — mounting a .dmg, copying an app to /Applications, ejecting a volume.

Reviewable by default

Risky actions pause for human approval. If the approval dialog can't appear, the run stops and writes a pending record you can clear from another terminal. Every decision lands in an append-only log.

Audit trail per run

Every run produces numbered screenshots, a markdown transcript, an approvals log, and a final report that separates "the model said it was done" from "the machine verified it" from "a human signed off".

02 / workflow

The loop, with the checkpoints that interrupt it.

The model never reaches a tool without passing through risk classification first. In the default mode, observe / low / medium actions pass through; high-risk actions ask for human approval; blocked actions return an error to the model so it can propose a safer alternative.

Why three checkpoints, not one

Plan approval catches "you misread the task". The per-action gate catches "the right plan, the wrong button". Final review catches "the model said it succeeded but it didn't". Different failure modes, different gates.

When the dialog can't show

Once the editor is hidden, terminal prompts are invisible. The per-action gate prefers a macOS dialog; if that times out it writes a pending record and surfaces an error to the model. Clear pendings later with the --approve-pending escape hatch.

03 / safety

Risk taxonomy, enforced in code.

Every tool call is classified before execution. The gate fires based on the review mode you chose. Blocked actions never execute, regardless of mode — they return a tool error to the model so it can try something safer.

Mac permissions check at startup

On launch, the agent probes whether macOS has authorized it to send approval dialogs. If not, you see the exact Settings path to fix it before the editor hides, not after.

FAILSAFE always on

Slam the cursor into any screen corner to abort the loop. The run's finally block still writes the final report so the audit trail isn't lost.

01 / architecture

Eight modules, layered top-down. Recorder is the substrate.

The agent loop drives downward: an entry point hands a task to the orchestrator, which calls Claude, classifies the proposed tool call, runs it, captures a screenshot, and repeats. Every action and every gate decision passes through recorder.py on its way to disk — the substrate at the bottom of the diagram. "REPL" below is short for read-eval-print loop — an interactive prompt that runs what you type and waits for the next line, like Python's >>> shell.

02 / module reference

One concern per file.

Slice-by-slice extraction from a single-file prototype. The orchestrator only knows about tool schemas and the loop; everything else is library code with its own tests.

agent.py

22 KB
Orchestrator

Parses args, hosts the computer + macos tool schemas, runs the planner, runs the loop, finally-writes the final report.

interactive.py

5.2 KB
REPL · interactive prompt

Lists recent runs with their final-report status. Dispatches /diagnose, /approve-pending, /list, /help, /quit.

review.py

22 KB
Gates · risk taxonomy

ReviewPolicy, classify_risk, the per-action gate, the TCC probe, the file fallback, the --approve-pending subcommand, the final report.

native_actions.py

6 KB
macOS host primitives

open_path, hide_vscode, install_app_from_dmg, eject_volume, verify_app_installed, run_applescript, restricted safe_shell.

screen.py

1.7 KB
Capture + downscale

screencapture at Retina, downscale to 1440×900 points, base64 encode for the API. Provides image_block / text_block.

recorder.py

1 KB · substrate
Numbered artifacts

Writes 001_initial.png … NNN_action.png plus a markdown transcript.md per run. One folder per run; the orchestrator and gates both write to it.

context_window.py

1.3 KB
API history compaction

compact_messages(messages, keep_last_images=N) keeps the N newest screenshots in API history, replaces older ones with text refs. Prevents 413 RequestTooLarge.

diagnostics.py

4.6 KB · off-loop
Preflight, no API calls

--diagnose writes display size, screencapture size, TCC state, mounted DMGs, whether VSCode can be hidden. Safe to run anytime.
