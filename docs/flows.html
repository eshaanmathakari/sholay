measured · cost · accuracy · tokens

Three deterministic flows, every run measured for cost, accuracy, and tokens.

The predecessor agent "navigated however it wanted." This layer inverts that: fixed, written-down playbooks across three application types — browser, no-API, legacy — each run scored by a per-flow oracle and recorded one row per run. It answers, per flow and per app-type: how reliable is it, what does a run cost, and how many tokens does it burn?

Live metrics dashboard
01 / what it is

A fixed playbook, executed by vision, measured at every step.

Each flow is a YAML playbook the agent executes one sub-goal at a time by looking at the screen — no scripted clicks. The runner reuses the agent's perception/action primitives so behaviour is identical, then records cost, tokens, accuracy, and timing for the run.

Fixed playbook

An ordered list of sub-goals in flows/*.yaml — same plan every run, live pixels. The LLM still executes by vision.

Measured every step

Per-step tokens, time, actions, and retries are captured as the run unfolds; cost is computed at verified $/MTok rates.

Scored by an oracle

A per-flow verifier turns the run into pass / fail / error — the independent answer key.

One row per run

No best-of-N. Every invocation appends one SQLite row; pass-rate and averages are queries over accumulated history.

02 / the key idea

The oracle is the answer key — machine where we can, human where we can't.

The oracle is independent of what the agent perceived, and its verification method is chosen to fit the app type. That single choice is what makes "no-API" a first-class app type rather than a gap.

no_api · demo
Human-verified

The playbook is fixed, but the agent never walks the exact same path twice: step count and wall-clock time drift run to run. 01 / architecture

Playbook → measured vision loop → oracle → metrics.

The runner loads a validated spec, drives the agent through it one sub-goal at a time, parses the final emitted JSON, and hands it to the flow's oracle. Cost, tokens, and status land in runs.db; report.py rolls it up into the dashboard.

02 / module reference

Drop a YAML + an oracle — no framework changes.

The runner, metrics store, pricing, and report are generic over flows. Flow #3 landed with zero edits to any of them.

runner.py

runner
Drives + measures

Step-by-step playbook execution, per-step token/time/action capture, emit parsing, oracle call, then the runs.db row + final_report.json.

flows/loader.py

spec
Validate / load

validate(dict) is pure (unit-testable without PyYAML); load(path) adds the file read. A malformed playbook fails loudly, never silently runs.

oracles/tradingview.py

oracle · machine
Independent quote

evaluate() pure; fetch_spx_quote() tries Yahoo → CNBC → stooq with retry. Quote-fetch failure → error, not a false fail.

oracles/proton.py

oracle · human
Approval gate

Reuses review.request_final_review(); the human's Approve / No / Feedback click becomes the status. fact_match is always None — no machine fact without an API.

metrics_db.py

store
SQLite, stdlib only

runs / step_metrics / feedback tables + aggregate(). Pass-rate is over scored runs; errors excluded from the denominator.

pricing.py

cost
Verified $/MTok

cost_usd(usage, model) on the verified claude-sonnet-4-6 rate table. Image/screenshot tokens are already inside the API counts.

report.py

report
CLI + dashboard

One query pass feeds a CLI table and docs/results.html — identical numbers, plus the architecture & cost narrative baked in.

feedback.py

capture-only
Human verdict

feedback.py <run_id> --pass/--fail --notes appends a verdict that may override the oracle, recorded alongside it — never mutating the run's status.

04 / permissions

What it takes to replicate — three TCC grants, no sudo.

The agent needs eyes, hands, and the ability to talk to apps — three macOS privacy permissions granted to the host terminal. Nothing privileged: the demo flows install nothing and safe_shell() hard-refuses sudo.

PermissionRoleWhy it's neededWithout it
Screen Recordingthe agent's eyesevery step shoots the screen via screencapture -x (screen.py)screenshots are black → run fails at once

no sudo native_actions.safe_shell() refuses sudo, rm, mv /, chmod/chown -R, diskutil erase, and anything outside a read-only allow-list. The only step that can ever ask for an admin password is the optional DMG-install helper for the not-yet-built legacy app #2 — never the browser or Proton flows.

First-run gotcha: macOS only prompts when a capability is first used, and a freshly granted Screen-Recording / Accessibility permission usually needs the terminal quit and reopened to take effect. Abort any live run by slamming the cursor into a screen corner (FAILSAFE).
