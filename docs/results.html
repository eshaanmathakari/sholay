Run-to-run variability

The playbook is fixed, but the agent never walks the exact same path twice — the screen shifts, the model picks differently, so step count and wall-clock time drift run to run. Each dot is one run; the solid tick is the median, the dashed tick the mean. When the mean sits to the right of the median the flow is right-skewed — a few long runs (the pre-fix TradingView thrash, a Proton dialog that waited on a human) drag the average up.

This report

Each flow is a fixed, written-down playbook the agent executes step-by-step by looking at the screen — no scripted clicks. Every run is scored by a per-flow oracle and recorded as one row: cost, tokens, accuracy.

The oracle is the independent answer key, chosen to fit the app type. Machine-verified where a source of truth exists — the browser flow checks the price the agent read off the chart against an independent market quote (±0.5%). Human-verified where none does — the no-API mail flow pops an approval dialog and the reviewer's click is the verdict. Both still record the same cost/token metrics.

How to read the tables: pass% is over scored runs only (pass ÷ (pass + fail)). An error means the oracle couldn't verify (unreachable source, dialog timeout) — shown separately as err, never counted as an accuracy failure. $/success = avg cost ÷ pass-rate is the headline efficiency metric.
